# ADR 002: Device Configuration Strategy

> Architectural decision for hardcoded device-to-SIP account mapping.

**Status**: DRAFT  
**Type**: Constraining  
**Date**: 2026-03-04  
**Source**: Legacy Analysis (`/legacy` command)

---

## Context

The gateway application runs on multiple Android devices, each requiring unique SIP account credentials and telephony configurations.

### Current Approach

Hardcoded device ID to configuration mapping in `Gateway.js`:

```javascript
let deviceId = DeviceInfo.getDeviceId();

if (deviceId == "sp7731cea") {
  tConfig = { ReplaceDialer: true, Permissions: true };
  sConfig = { name: "50015", username: "50015", password: "pass50015" };
} else if (deviceId == "Impress_City") {
  tConfig = { ReplaceDialer: false, Permissions: false };
  sConfig = { name: "50014", username: "50014", password: "pass50014" };
}
// ... more devices
```

### Device Mapping Table

| Device ID | Device Name | SIP Account | ReplaceDialer | Permissions | FAS |
|-----------|-------------|-------------|---------------|-------------|-----|
| `sp7731cea` | Unknown | 50015 | true | true | - |
| `Impress_City` | Unknown | 50014 | false | false | - |
| `Impress_Tor` | Unknown | 50017 | true | true | - |
| `QC_Reference_Phone` | Redmi 4A | 50019 | true | true | false |
| `MSM8937` | Redmi 5A | 50018 | true | true | false |
| Default | - | 50016 | false | false | - |

---

## Decision

**Hardcoded Configuration**: Device-specific settings embedded in source code.

### Configuration Parameters

#### Telephony Configuration
- `ReplaceDialer`: Replace default dialer app (boolean)
- `Permissions`: Request elevated permissions (boolean)
- `fas`: Disable FAS/VoLTE (boolean, device-specific)

#### SIP Configuration
- `name`: SIP account display name
- `username`: SIP username for authentication
- `password`: SIP password (hardcoded)

### Configuration Flow

```
Gateway.start()
│
├─> Get: deviceId = DeviceInfo.getDeviceId()
│
├─> Match: Device ID → Configuration
│   ├─ Known device: Use specific config
│   └─ Unknown device: Use fallback (50016)
│
├─> Initialize: tEndpoint with tConfig
└─> Initialize: sEndpoint with sConfig
```

---

## Consequences

### Positive

✅ **Simplicity**: No external config files or APIs needed  
✅ **Performance**: Instant configuration lookup (no I/O)  
✅ **Version Control**: Configuration changes tracked with code  
✅ **Device Isolation**: Each device has unique SIP account  

### Negative

❌ **Scalability**: Adding devices requires code changes and redeployment  
❌ **Security**: SIP passwords visible in source code  
❌ **Maintenance**: Device IDs may change with firmware updates  
❌ **Flexibility**: Cannot update config without app update  

### Security Concerns

1. **Credential Exposure**: Passwords hardcoded (`pass50015`, etc.)
2. **No Encryption**: Credentials stored in plain text
3. **Source Control Risk**: Passwords committed to repository

---

## Alternatives Considered

### 1. External Configuration File
```
/config/devices.json
{
  "sp7731cea": { "sip": { "username": "50015", "password": "..." } }
}
```
**Rejected**: Still exposes credentials, adds I/O complexity

### 2. Remote Configuration API
```
GET /api/device-config?deviceId=sp7731cea
```
**Rejected**: Requires network connectivity, adds dependency

### 3. Environment Variables
```
SIP_USERNAME=50015
SIP_PASSWORD=pass50015
```
**Rejected**: Android app context, not practical for React Native

### 4. Encrypted Storage
```
React-Native-Keystore → Decrypt credentials at runtime
```
**Rejected**: Adds complexity, still needs initial key

---

## Recommendations

### Short-Term
1. **Remove passwords from source**: Use placeholder or build-time injection
2. **Add device documentation**: Document why each device needs specific config
3. **Validate device IDs**: Add logging for unknown devices

### Long-Term
1. **Migrate to dynamic config**: External config file or API
2. **Implement credential encryption**: Use Android Keystore
3. **Add config validation**: Verify SIP credentials on app startup

# ADR 001: Unidirectional Bridge Architecture

> Architectural decision for SIP→GSM only bridging with incoming GSM call rejection.

**Status**: DRAFT  
**Type**: Constraining  
**Date**: 2026-03-04  
**Source**: Legacy Analysis (`/legacy` command)

---

## Context

The Telon GSM-SIP Gateway needs to enable SIP-based call routing through GSM hardware while maintaining security and preventing unauthorized access.

### Requirements

1. Allow SIP calls to trigger GSM calls (gateway functionality)
2. Prevent unauthorized calls through the gateway
3. Maintain call state synchronization between protocols
4. Support multiple devices with different configurations

### Current Implementation

```javascript
// Gateway.js
const sip2tele = true;   // SIP → GSM: ENABLED
const tele2sip = false;  // GSM → SIP: DISABLED

// Incoming GSM calls automatically rejected
onTeleCallReceived = (call) => {
  if (call._state == "TELE_INV_STATE_RINGING") {
    this.tEndpoint.hangupCall(call);  // Reject immediately
  }
}
```

---

## Decision

**Unidirectional Bridge**: Only SIP→GSM call initiation is enabled.

### Architecture

```
SIP Network          Gateway            GSM Network
    │                   │                    │
    │── Call Received ──│                    │
    │                   │── Make Call ───────│→
    │                   │                    │
    │← State Updates ───│── State Sync ──────│
    │                   │                    │
    │── Hangup ─────────│── Hangup ──────────│→
    │                   │                    │
    │                   │← Incoming Call ────│
    │                   │── REJECT ──────────│
```

### Security Layers

1. **Incoming Call Rejection**: All incoming GSM calls automatically rejected
2. **GSM→SIP Disabled**: `tele2sip=false` prevents GSM-initiated SIP calls
3. **Device Binding**: Each device has unique SIP account (50014-50019)
4. **Root Access**: Magisk module for system-level permissions

---

## Consequences

### Positive

✅ **Security**: Prevents unauthorized calls through gateway  
✅ **Simplicity**: Unidirectional flow is easier to reason about  
✅ **Control**: All calls originate from SIP infrastructure  
✅ **Audit Trail**: SIP server maintains call logs  

### Negative

❌ **Limited Functionality**: Cannot initiate SIP calls from GSM  
❌ **Root Requirement**: Magisk dependency limits deployment  
❌ **Single Point of Failure**: SIP server outage blocks all calls  

### Risks

1. **Hangup Flag Bug**: `sendHangup` flag never reset (could block legitimate hangups)
2. **Credential Exposure**: SIP passwords hardcoded in source
3. **State-Based Detection**: Incoming call detection via state (less reliable than direction-based)
